# Week 7: RAG Security Knowledge Assistant — Evaluation Report

## 1. Setup Summary

- **LLM:** Llama-3.3-70b-versatile via Groq (temperature: 0.3, streaming enabled)
- **Embeddings:** sentence-transformers/all-MiniLM-L6-v2 via HuggingFace Inference API
- **Vector Store:** In-Memory Vector Store (Top K: 4)
- **Text Splitter:** Recursive Character Text Splitter (chunk size: 1000, chunk overlap: 200)
- **Documents loaded:**
  1. `mitre-initial-access.txt` — MITRE ATT&CK Tactic TA0001: Initial Access (11 techniques including Phishing sub-techniques, Supply Chain Compromise, Valid Accounts) — approximately 3 pages
  2. `mitre-credential-access.txt` — MITRE ATT&CK Tactic TA0006: Credential Access (17 techniques including Brute Force, OS Credential Dumping, Kerberos Tickets, Unsecured Credentials) — approximately 4 pages
  3. `mitre-lateral-movement.txt` — MITRE ATT&CK Tactic TA0008: Lateral Movement (9 techniques including Remote Services, Pass the Hash, Remote Service Session Hijacking) — approximately 3 pages

## 3. Edge Case Observations

- **Unrelated question ("What is the weather like today?"):** The chatbot responded with "Hmm, I'm not sure," which is the ideal behavior for a well-constrained RAG system. Notably, the vector store still returned its Top K chunks (T1072 - Software Deployment Tools, T1563 - Remote Service Session Hijacking), but the LLM correctly recognized these were irrelevant to the weather question and chose not to fabricate an answer from them. This demonstrates that the Conversational Retrieval QA Chain properly constrains the LLM when questions fall entirely outside the knowledge base domain.

- **Related topic not in documents ("What are the latest CVEs from 2026?"):** The chatbot again responded "Hmm, I'm not sure." This is a stronger test because the question is cybersecurity-related, meaning the retrieved chunks had higher semantic similarity to the query than the weather question. The returned source chunks (.002 - Compromise Software Supply Chain, .001 - Compromise Software Dependencies, T1080 - Taint Shared Content) are tangentially related to vulnerabilities but the LLM correctly identified that none contained CVE-specific information and declined to answer. This confirms the RAG system constrains even domain-adjacent questions.

- **Closely related tactic not uploaded ("What techniques do adversaries use for data exfiltration?"):** The chatbot hallucinated a detailed, plausible-sounding answer listing Exfiltration Over Alternative Protocol, Exfiltration Over Physical Medium, Data Encoding, Data Compression, and Data Encryption — none of which exist in any of the three uploaded documents. The response began with "According to the provided context," falsely attributing fabricated information to the retrieved documents. The source chunks shown (.002 - Compromise..., Adversaries may...) were clearly unrelated to exfiltration. This is the most dangerous type of RAG hallucination because the question falls within the same MITRE ATT&CK domain as the uploaded documents, and the LLM's training data contains extensive knowledge of ATT&CK techniques. The high domain proximity gave the LLM enough confidence to generate an authoritative-sounding answer rather than admitting uncertainty. In a production system, this could be mitigated with stricter system prompts instructing the LLM to only answer when retrieved chunks directly address the question, or by implementing a minimum similarity score threshold on the retriever.

## 4. Settings Experiments

Settings experiments were not performed during this lab session due to time constraints. The default configuration (temperature 0.3, chunk size 1000, chunk overlap 200, Top K 4) produced strong results for most test cases, suggesting these are reasonable defaults for a cybersecurity knowledge assistant. Future experimentation could explore reducing temperature to 0.1 to further constrain hallucination on broad questions like Q5, increasing Top K to 6 for better cross-document retrieval, or testing smaller chunk sizes (500) to improve precision on specific technique lookups.

## 5. Reflection

**What surprised me about how RAG works:** The most surprising finding was the inconsistency of hallucination behavior across the three edge case questions. The chatbot correctly admitted uncertainty for completely unrelated questions (weather) and even domain-adjacent questions (CVEs), but hallucinated confidently when asked about a closely related MITRE ATT&CK tactic (exfiltration) that was not in the knowledge base. This revealed that hallucination risk in RAG is not binary — it exists on a spectrum determined by the semantic proximity between the question and the uploaded documents. The closer the question is to the domain of the knowledge base, the more likely the LLM is to blend its training data with retrieved context and produce a convincing but unsupported answer.

**How I could improve this chatbot for real-world use:** Several improvements would make this chatbot production-ready. First, replacing the In-Memory Vector Store with a persistent database like Pinecone or Chroma would eliminate the need to re-index documents after every session restart. Second, adding a stricter system prompt that explicitly instructs the LLM to say "I don't have information on that topic" when retrieved chunks don't directly address the question would reduce domain-adjacent hallucinations. Third, implementing a minimum similarity score threshold on the retriever would prevent irrelevant chunks from being passed to the LLM at all. Finally, expanding the knowledge base to cover all 14 MITRE ATT&CK Enterprise tactics (not just 3) and the full NIST CSF 2.0 framework would dramatically increase the chatbot's coverage and reduce the surface area for hallucination.

**How I might use RAG in my capstone project:** My capstone project centers on an Email Triage and Auto-Responder system. RAG could be integrated as a knowledge base component that provides the system with organizational policy documents, standard operating procedures, and escalation guidelines. When the system triages an incoming email, it could query a RAG pipeline to retrieve relevant policy context before drafting a response — ensuring that auto-generated replies align with organizational standards rather than relying solely on the LLM's general training data. This would be especially valuable for handling sensitive or compliance-related emails where accuracy and policy adherence are critical. The Flowise chatflow built in this lab could serve as the starting prototype, with the MITRE ATT&CK documents replaced by organizational policy documents and the Flowise API called from the n8n automation workflow built in earlier labs.
